| 项目        | 配置                                          |
| ----------- | --------------------------------------------- |

| CPU         | Intel Core i9-14900HX，16 cores               |

本次测试在 WSL2 环境下进行，测试结果主要用于验证当前版本的吞吐能力、延迟表现、协议正确性和异常场景稳定性。

## 2. 测试方法

本项目是基于 epoll 的 C++ TCP 长连接消息接入服务，不是 HTTP Server，因此本次压测采用长连接模型：

1. 每个 client 建立一条 TCP 连接。
2. 每条连接连续发送多个请求。
3. 每个请求都按照项目自定义二进制协议编码。
4. 每个响应都按照自定义协议解码。
5. 客户端校验响应中的 `request_id` 是否和请求匹配。
6. 所有请求完成后统计 QPS、平均延迟、P50、P95、P99 和最大延迟。

压测脚本使用：

| 脚本                             | 作用                              |
| -------------------------------- | --------------------------------- |
| `scripts/benchmark.py`           | ECHO / LOG_PUSH 性能压测          |
| `scripts/test_half_packet.py`    | 半包测试                          |
| `scripts/test_sticky_packet.py`  | 粘包测试                          |
| `scripts/test_invalid_length.py` | 非法 body_length 测试             |
| `scripts/test_slow_client.py`    | 慢客户端 / output_buffer 上限测试 |

构建方式：

    cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
    cmake --build build -j

启动服务端：

    cd build
    ./message_server

## 3. 指标说明

| 指标            | 含义                                     |
| --------------- | ---------------------------------------- |
| Clients         | 并发 TCP 连接数                          |
| Reqs/Client     | 每个连接发送的请求数量                   |
| Total           | 总请求数，等于 `Clients × Reqs/Client`   |
| Success         | 成功收到响应，并且 `request_id` 匹配     |
| Failed          | 请求失败、超时、协议解析失败或响应不匹配 |
| QPS             | 每秒成功处理的请求数                     |
| Avg Latency     | 平均往返延迟，单位 ms                    |
| P50 / P95 / P99 | 延迟分位数，单位 ms                      |
| Max Latency     | 最大单次请求延迟，单位 ms                |

## 4. ECHO 压测结果

ECHO 场景主要验证网络收发、协议解析、任务分发、响应写回和 `request_id` 匹配能力。

| 场景 | Clients | Reqs/Client | Total | Success | Failed |    QPS | Avg (ms) | P50 (ms) | P95 (ms) | P99 (ms) | Max (ms) |
| ---- | ------: | ----------: | ----: | ------: | -----: | -----: | -------: | -------: | -------: | -------: | -------: |
| A    |       1 |        1000 |  1000 |    1000 |      0 |    9.9 |   100.76 |   100.77 |   101.54 |   103.18 |   103.67 |
| B    |      10 |        1000 | 10000 |   10000 |      0 | 4664.2 |     1.50 |     1.39 |     2.10 |     2.50 |   101.08 |
| C    |     100 |         100 | 10000 |   10000 |      0 | 2945.6 |     7.17 |     7.50 |    13.79 |    18.19 |   100.62 |
| D    |     300 |          50 | 15000 |   15000 |      0 | 5063.0 |    27.55 |    28.04 |    50.13 |    62.95 |   100.80 |
| E    |     500 |          20 | 10000 |   10000 |      0 | 6913.8 |     9.69 |     9.51 |    15.14 |    18.24 |    30.06 |

### 结果分析

单连接场景下 QPS 较低，平均延迟约为 100 ms。主要原因是当前服务端的 `response_queue` 是在 `epoll_wait` 返回后统一 drain 的，而 `epoll_wait` timeout 设置为 100 ms。在单连接低并发场景下，响应可能需要等待下一轮 epoll timeout 才被写回，因此出现了明显的 100 ms 延迟。

当并发连接数提升到 10 后，QPS 提升到 4664.2，平均延迟下降到 1.50 ms。原因是多连接并发会持续触发 epoll 事件，主事件循环更加频繁地处理 `response_queue`，100 ms timeout 不再成为主要瓶颈。

在 100、300、500 并发场景下，服务端均能完成全部请求处理，没有失败请求、没有超时请求、没有 `request_id` 错配。说明当前版本在中等并发长连接场景下具备较好的稳定性。

## 5. LOG_PUSH 压测结果

LOG_PUSH 场景用于验证日志 / 消息接入能力。相比 ECHO，LOG_PUSH 会进行 JSON 解析、字段校验、日志写入和 ACK 响应。

| Clients | Reqs/Client | Payload Size | Total | Success | Failed |    QPS | Avg (ms) | P50 (ms) | P95 (ms) | P99 (ms) | Max (ms) |
| ------: | ----------: | -----------: | ----: | ------: | -----: | -----: | -------: | -------: | -------: | -------: | -------: |
|     100 |         100 |        257 B | 10000 |   10000 |      0 | 6354.4 |    13.35 |    13.20 |    20.16 |    25.33 |   100.33 |

### 结果分析

本次 LOG_PUSH 在 100 并发、总计 10000 请求下全部成功，失败数为 0。

LOG_PUSH 的 QPS 为 6354.4，高于同等 100 并发下的 ECHO 场景。主要原因是 LOG_PUSH 的响应是固定较小的 ACK，而 ECHO 会把 payload 回传给客户端，响应体更大。因此在当前测试规模下，LOG_PUSH 的 JSON 解析和文件写入没有成为主要瓶颈。

需要注意的是，`logs/access.log` 是运行时日志文件，项目当前不会在每次压测前自动清空它。因此统计日志写入数量时，不能简单长期依赖文件总行数。更严谨的做法是：

1. 压测前清空 `logs/access.log`；
2. 运行 LOG_PUSH 压测；
3. 压测后统计新增日志行数；
4. 确认新增行数与成功 LOG_PUSH 请求数一致。

在本次测试中，LOG_PUSH 成功响应数量为 10000，可以说明服务端完成了 10000 条日志接入请求。

## 6. 稳定性测试结果

除了常规性能压测，本次还验证了 TCP 流式协议中常见的异常场景。

| 测试项       | 预期行为                                                | 实际结果 |
| ------------ | ------------------------------------------------------- | -------- |
| 半包测试     | 服务端等待更多数据，完整包到达后再解析                  | PASSED   |
| 粘包测试     | 一次 `sendall` 发送多个包，服务端能连续解析多个请求     | PASSED   |
| 非法长度测试 | 对非法 `body_length` 主动关闭连接，服务端不崩溃         | PASSED   |
| 慢客户端测试 | 客户端不读响应时，服务端在 output_buffer 超限后关闭连接 | PASSED   |

### 额外验证

压测和异常测试结束后，服务端仍然可以响应新的 PING 请求，说明服务端没有因为异常输入、慢客户端或大量请求而崩溃。

本次测试也没有发现明显的连接泄漏、`request_id` 错配或服务端进程异常退出问题。

## 7. request_id 正确性

本次所有 ECHO 和 LOG_PUSH 压测请求均进行了 `request_id` 校验。

测试结果显示：

| 类型     | 总请求数 | request_id 错配 |
| -------- | -------: | --------------: |
| ECHO     |    46000 |               0 |
| LOG_PUSH |    10000 |               0 |
| 总计     |    56000 |               0 |

这说明当前版本中：

1. 响应能够正确关联原始请求。
2. worker 线程处理后返回的响应没有串到其他连接。
3. fd 复用保护逻辑有效，没有出现旧连接响应写到新连接的问题。
4. 多连接并发情况下没有发生响应错配。

## 8. 已知瓶颈

### 8.1 response_queue 唤醒机制

当前 `response_queue` 的处理依赖主事件循环在 `epoll_wait` 返回后统一 drain。由于 `epoll_wait` timeout 为 100 ms，低并发场景下会出现接近 100 ms 的额外响应延迟。

这个问题在单连接压测中非常明显：

| Clients | Avg Latency |
| ------: | ----------: |
|       1 |   100.76 ms |
|      10 |     1.50 ms |

后续可以通过 `eventfd` 优化：worker 线程把响应放入 response queue 后，通过 `eventfd` 唤醒 epoll 主循环，让响应可以立即被处理，而不是等待 timeout。

### 8.2 单 Reactor 事件循环

当前主 IO 线程负责 accept、read、write、epoll event 处理和 response queue drain。中低并发场景下足够使用，但在更高并发连接数下，单个 Reactor 线程可能成为瓶颈。

后续可以考虑：

1. 主 Reactor 只负责 accept；
2. 多个 sub Reactor 负责连接 IO；
3. worker pool 负责业务处理；
4. response 使用 eventfd 或 lock-free queue 唤醒对应 Reactor。

### 8.3 LogStorage 文件写入

LOG_PUSH 当前会写入本地 `access.log`，并通过 mutex 保护文件写入。当前测试规模下没有成为主要瓶颈，但在更高 QPS 下，文件锁和同步写入可能限制吞吐。

后续可以考虑：

1. 批量写日志；
2. 定时 flush；
3. 单独日志线程；
4. 使用异步队列解耦业务处理和文件写入。

### 8.4 Python 压测客户端限制

本次压测脚本使用 Python 编写，并且主要依赖多线程模拟并发连接。Python 线程模型和 GIL 可能会在高 QPS 场景下成为客户端侧瓶颈。

因此当前 QPS 数字只能说明在该测试环境和该测试脚本下的表现，不应作为服务端极限性能结论。

如果后续要测试服务端极限吞吐，可以考虑使用 C++ / Go 编写更高性能的压测客户端。

### 8.5 Buffer 内存复用

当前连接 input/output buffer 使用字符串或动态缓冲区保存数据。大 payload 或慢客户端场景后，buffer capacity 可能不会立即释放。虽然目前已经有 output_buffer 上限保护，但长期运行时仍可以进一步优化 buffer 复用策略。

---

## 9. 后续优化方向

1. 使用 `eventfd` 唤醒 epoll，降低低并发场景下的响应延迟。
2. 对 LOG_PUSH 增加批量写入，减少频繁加锁和 flush。
3. 增加更详细的服务端指标统计，例如 active connections、total requests、failed requests、bytes read/write。
4. 支持连接级别的读写水位线，防止单个慢连接占用过多内存。
5. 增加更高并发压测，例如 1000、3000、5000 长连接。
6. 使用 C++ 或 Go 编写高性能 benchmark client，降低 Python 客户端瓶颈影响。
7. 使用 `perf` / flamegraph 分析 CPU 热点。
8. 对 buffer 做复用或池化，降低频繁分配带来的性能损耗。

---

## 10. 结论

本次压测验证了当前版本在中等并发长连接场景下的可用性和稳定性。

主要结论：

1. ECHO 场景下，服务端完成 46000 次请求，失败数为 0。
2. LOG_PUSH 场景下，服务端完成 10000 次日志接入请求，失败数为 0。
3. 所有压测请求均未出现 `request_id` 错配。
4. 半包、粘包、非法长度和慢客户端测试均通过。
5. 压测结束后，服务端仍然可以响应新请求。
6. 当前最大明显瓶颈是低并发场景下 `response_queue` 依赖 epoll timeout drain，后续可通过 `eventfd` 优化。

整体来看，当前版本已经具备较好的协议正确性、连接稳定性和基础并发处理能力，可以作为后续继续优化性能和扩展业务功能的基础版本。
